# Graft: A Package Manager for Config Files

**Date:** 2026-03-14
**Status:** draft

## Problem

Developers copy files between projects constantly — Makefiles, GitHub Actions workflows, Claude Code skills, CI configs, linter configs. Tools like diecut solve the macro version (scaffold a whole project from a template), but there's no good solution for the micro version: depending on individual files or small bundles that evolve over time and pulling updates.

The gap is a **versioned, ongoing relationship** with upstream files — not a one-time copy.

## Design

### Core Concepts

- **Source** — A file or directory in a git repo, referenced as `gh:owner/repo/path`. Nothing special needed to publish: just files in a repo with git tags (or commit SHAs).
- **Graft** — A declared dependency in the project's `graft.toml`. Maps a source to a local destination path. The relationship between "that upstream path" and "this local path." Named after the horticultural practice of attaching a living branch from one tree onto another, where it continues to grow.
- **Manifest (`graft.toml`)** — User-edited file declaring what you depend on. Contains source, destination, and version. Checked into the repo.
- **Lockfile (`graft.lock`)** — Auto-generated file recording the exact resolved state: precise commit SHAs, checksums, and file lists. Checked into the repo. Never hand-edited.

### Graft States

- **Synced** — Local file matches the locked version (checksum matches lockfile).
- **Modified** — Local file has been edited since last sync (checksum differs from lockfile).
- **Outdated** — Upstream has a newer version available, and the file content at the new tag actually differs from the locked version (tag bumps where the grafted file didn't change are not reported as outdated).
- **Conflicted** — Upgrade attempted, three-way merge produced conflicts. User must resolve and run `graft resolve`.

### Manifest Format (`graft.toml`)

The manifest is what the user writes and maintains. It declares intent.

```toml
[deps.ci-lint]
source = "gh:rroskam/shared-configs/workflows/lint.yml"
version = "v1.2.0"
dest = ".github/workflows/lint.yml"

[deps.makefile]
source = "gh:rroskam/shared-configs/Makefile"
version = "v1.2.0"
dest = "Makefile"

# Graft to a specific commit (no tag required)
[deps.eslint-config]
source = "gh:company/internal-configs/.eslintrc.json"
version = "a1b2c3d"
dest = ".eslintrc.json"

# A bundle — multiple files from one source directory
[deps.claude-skills]
source = "gh:rroskam/shared-configs/skills/"
version = "v2.0.0"
dest = ".claude/skills/"
files = ["brainstorming.md", "debugging.md", "tdd.md"]
```

- `source`: `gh:owner/repo/path` shorthand (version is always in the separate `version` field, not in the source string). Full URLs also supported. Works with both public and private repos.
- `version`: Git tag or short commit SHA. When a tag, semver tags are sorted correctly; non-semver compared lexicographically. When a SHA, grafted to that exact commit (no "newer version" detection).
- `dest`: Local path (file or directory) where the content is placed. Must be within the project root — paths containing `..` that escape the project directory or target `.git/` are rejected.
- `files` (optional): For directory sources, explicitly list which files to include. If omitted, all files in the directory are included.
- Duplicate `dest` values across grafts are a validation error.

### Lockfile Format (`graft.lock`)

The lockfile is auto-generated by `graft sync` and `graft upgrade`. It records the exact resolved state. TOML format, consistent with the manifest.

```toml
[deps.ci-lint]
source = "gh:rroskam/shared-configs/workflows/lint.yml"
version = "v1.2.0"
commit = "f4e5d6c7a8b9..."
checksum = "sha256:abc123..."

[deps.makefile]
source = "gh:rroskam/shared-configs/Makefile"
version = "v1.2.0"
commit = "f4e5d6c7a8b9..."
checksum = "sha256:def456..."

[deps.eslint-config]
source = "gh:company/internal-configs/.eslintrc.json"
version = "a1b2c3d"
commit = "a1b2c3d4e5f6..."
checksum = "sha256:789abc..."

[deps.claude-skills]
source = "gh:rroskam/shared-configs/skills/"
version = "v2.0.0"
commit = "1a2b3c4d5e6f..."
checksum = "sha256:aaa111..."
files = ["brainstorming.md", "debugging.md", "tdd.md"]
```

- `commit`: Full commit SHA that the version resolved to. This is the immutable anchor — even if a tag is force-pushed, the lockfile records what was actually fetched.
- `checksum`: SHA-256 hash of the file as originally installed. For directories: sort filenames by full relative path lexicographically, concatenate each file's SHA-256 hash, then hash the result. Used to detect local modifications.
- `files`: For directory grafts, the resolved list of files (may differ from manifest if manifest omits `files` and includes all).

### CLI Commands

- **`graft init`** — Create an empty `graft.toml` with a comment explaining the format. If `graft.toml` already exists, no-op.
- **`graft add <source@version> <dest> [--adopt]`** — Add a dependency to `graft.toml` and `graft.lock`.
  - **Without `--adopt`:** Fetches the upstream file and writes it to `dest`. The lockfile checksum records the fetched content. If `dest` already exists, add refuses (use `--adopt` or `--force` to override).
  - **With `--adopt`:** For files that already exist locally. Adds the graft without overwriting the local file. Fetches the upstream version at the specified tag/SHA to record the base in `graft.lock`, but preserves the local file as-is. If the local file differs from upstream, the graft immediately enters the "modified" state — which means the first `graft upgrade` will do a proper three-way merge between the version you claimed as the base, your current file, and the latest upstream. This is how you adopt graft for files you've already copied and possibly modified.
- **`graft sync`** — Resolve and fetch all dependencies per `graft.toml`, update `graft.lock`. Only writes missing files — if a file already exists on disk, sync skips it. Use `graft upgrade` to reconcile existing files with newer versions.
- **`graft check`** — Verify that all grafts are in a clean state. Exits with code 0 if everything is synced, code 1 if any grafts are modified, outdated, or conflicted. Designed for CI pipelines.
- **`graft outdated`** — Show grafts where upstream has newer versions than what's in `graft.lock`. Read-only. SHA-grafted deps are skipped (no automatic discovery). Uses smart detection: compares file content at the new tag against the locked checksum, so tag bumps where the grafted file didn't actually change are not reported.
- **`graft upgrade [name] [--dry-run]`** — With a name: fetch new upstream version and three-way merge if locally modified. Without a name: upgrade all outdated grafts. With `--dry-run`: show what would change without modifying any files.
- **`graft list`** — Show all grafts with status (synced, modified, outdated, conflicted), including version numbers and destination paths.
- **`graft resolve <name>`** — After manually resolving conflict markers, run this to update `graft.lock` with the new version and checksum of the resolved file.
- **`graft remove <name>`** — Remove from `graft.toml` and `graft.lock`. Leave local file in place (it's yours now).

### Three-Way Merge

When `graft upgrade <name>` encounters a locally modified file:

1. **Base** — Fetch file at the commit recorded in `graft.lock` (the exact version originally installed).
2. **Ours** — Current local file on disk.
3. **Theirs** — File at the new upstream version.
4. If base == ours (never customized): overwrite with theirs. Update `graft.lock` with new version, commit, and checksum.
5. If base != ours: run three-way merge via `git merge-file`.
   - **Clean merge:** Auto-apply. Update `graft.lock` with new version, commit, and checksum of merged file.
   - **Conflicts:** Write conflict markers to file. Set graft state to conflicted. Do NOT update `graft.lock` version or checksum until user runs `graft resolve`.

All file writes (merged files, `graft.lock`) use atomic write-to-temp-then-rename to prevent corruption on crash or interruption.

For directories: merge is per-file. New upstream files are added. Removed upstream files are reported as warnings on stdout but not auto-deleted.

### Source Resolution & Authentication

- `gh:owner/repo/path` is shorthand expanded to GitHub's API.
- **Single-file grafts:** Use the GitHub Contents API (`GET /repos/{owner}/{repo}/contents/{path}?ref={ref}`) to fetch individual files without downloading the entire repo tarball.
- **Directory grafts:** Use GitHub's tarball API with streaming extraction — download the tarball, iterate tar entries, extract only matching paths, discard the rest.
- **Authentication:** Uses the same git credential chain the user already has configured. Resolution order for API calls: (1) `GH_TOKEN` / `GITHUB_TOKEN` env var, (2) `gh auth token` output if `gh` CLI is installed, (3) unauthenticated (public repos only). No custom auth configuration.
- **Private repos:** Work transparently as long as the user has credentials configured via any of the above mechanisms.
- Fetched content cached in `~/.graft/cache/` keyed by source + commit SHA (not tag, since tags can be force-pushed). `graft sync --no-cache` bypasses the cache if needed. Cache writes use atomic temp-then-rename.
- **Version discovery:** `git ls-remote --tags` for discovering available versions during `graft outdated` and `graft upgrade`. Annotated tags produce both a tag-object entry and a dereferenced `^{}` entry — always use the `^{}` SHA (the actual commit) when present. For SHA-grafted deps, no upgrade discovery.

### Version Resolution

- **Tag versions** (e.g., `v1.2.0`): Resolved to the commit the tag points to. `graft upgrade` discovers newer tags via `git ls-remote --tags` and uses semver sorting when tags follow semver. Strip leading `v` before semver parsing.
- **SHA versions** (e.g., `a1b2c3d`): Resolved to the full commit SHA. No automatic upgrade discovery — the user explicitly chose a point in time. `graft upgrade` skips these unless the user manually updates the version in `graft.toml`.
- **Detection:** If `version` matches `^[0-9a-f]{7,64}$`, it's treated as a SHA. Otherwise it's treated as a tag/ref. (Git SHA-1 hashes are 40 hex chars; Git SHA-256 hashes are 64. Short SHAs down to 7 chars are accepted as input but always resolved and stored as full SHAs in `graft.lock`.)

### Error Handling

The tool must provide clear, actionable error messages for common failure modes:

- **Auth failure:** "Could not authenticate to `gh:owner/repo`. Set `GH_TOKEN`, run `gh auth login`, or verify the repo is public."
- **Source path not found:** "Path `workflows/lint.yml` not found in `owner/repo` at version `v1.2.0`."
- **Version not found:** "Tag `v1.2.0` not found in `owner/repo`. Available tags: v1.1.0, v1.0.0."
- **No manifest:** Running any command without `graft.toml` suggests `graft init`.
- **Network failure during upgrade:** Leave all files untouched. "Network error fetching `owner/repo`. No files were modified."
- **Dest already exists on add:** "File `Makefile` already exists. Use `--adopt` to track the existing file, or `--force` to overwrite."

## Implementation

- **Language:** Rust (consistent with diecut, single static binary). Requires `git` on PATH for `git merge-file` and `git ls-remote`.
- **Project location:** `~/repos/graft`, scaffolded from diecut's project structure.
- **HTTP client:** `reqwest` with `rustls-tls`. Use blocking client for v1 (no async complexity needed without concurrent fetches).
- **Git fetching:** GitHub Contents API for single files, tarball API for directories. `git ls-remote` for version discovery.
- **Merge:** Shell out to `git merge-file` with temp files for base and theirs. Ours is the working copy.
- **File writes:** Atomic write-to-temp-then-rename for `graft.lock`, cache entries, and merged files. Use `tempfile` crate with `NamedTempFile::persist()`.
- **Relationship to diecut:** Separate tool, separate repo. May share `gh:` source resolution as a crate later, no coupling for v1.

## Out of Scope for v1

- **Parameterization / templating** — Fetch as-is, edit locally if needed.
- **Transitive dependencies** — Grafts don't depend on other grafts.
- **Version constraints / ranges** — v1 grafts to exact tags or SHAs only. Semver range operators (`^1.2.0`, `~1.2.0`) are a planned future addition. The `version` field is designed for backwards compatibility: a bare version like `v1.2.0` will continue to mean "exactly this version" when ranges are added. Ranges would enable a `graft update` command (resolve within constraints) alongside the existing `graft upgrade` (bump beyond constraints), matching the cargo model.
- **Auto-sync / hooks** — Explicit `graft sync` only. A reference GitHub Action for scheduled `graft check` is a fast-follow.
- **Registry / discovery** — No search or index. Find packages via READMEs and GitHub search.
- **Conflict resolution UI** — Standard git conflict markers.
- **Publishing workflow** — Tag your repo. That's it.
- **Non-GitHub sources** — v1 is GitHub-only (covers both public and private). GitLab, Bitbucket, etc. can come later.
- **Sub-file granularity** — v1 operates at file/directory level. Section-level grafting (e.g., pinning sections of a CLAUDE.md) is a potential future direction.
